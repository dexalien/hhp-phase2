# HHP On-Chain Flow

## For the judges (pitch)

Hacker House Protocol uses Account Abstraction (ERC-4337) and smart contracts on Arbitrum so builders can book and pay for spots in Hacker Houses **without exposing their personal wallet** and without any centralized platform holding their funds.

> **One line:** A builder pays for a spot with USDC, signs from their protected identity, receives an NFT as proof of reservation, and their funds sit in an on-chain escrow until the host releases them — no intermediaries.

---

## Addresses, Wallets & Transactions — Field Guide

This section documents every concept you encounter when exploring HHP on-chain: wallet types, hash types, and why some things don't show up in Arbiscan.

---

### The three identity layers in HHP

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: Embedded Wallet (EOA)                             │
│  Generated by Privy at sign-up                              │
│  Address: 0x8D075aa7...                                     │
│  Role: SIGNER — holds the cryptographic key                 │
│  On-chain: NEVER appears directly                           │
│  Visible in: "Connected Wallet" inside the app              │
└──────────────────────────┬──────────────────────────────────┘
                           │ signs UserOperations
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  LAYER 2: Smart Wallet / Kernel (ERC-4337)                  │
│  Deployed by ZeroDev on first transaction                   │
│  Address: 0x40c8aEc3...                                     │
│  Role: ON-CHAIN ACTOR — pays, receives NFTs, calls contracts│
│  On-chain: YES, visible on Arbiscan                         │
│  This is "your wallet" from the contracts' perspective      │
└──────────────────────────┬──────────────────────────────────┘
                           │ UserOps route through
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  LAYER 3: EntryPoint ERC-4337                               │
│  Global standard contract                                   │
│  Address: 0x4337009B...                                     │
│  Role: EXECUTOR — receives UserOps from the bundler         │
│        and executes them on-chain                           │
│  On-chain: appears as "caller" in many internal calls       │
└─────────────────────────────────────────────────────────────┘
```

**The golden rule:** what the user signs is a UserOperation, not an Ethereum transaction. The actual Ethereum transaction is created by the **bundler** (ZeroDev) — it pays its own gas and submits it to the EntryPoint on your behalf.

### Hash types — how to tell them apart

Every action in HHP (mint, pay, etc.) produces a hash. They all look the same visually (`0x` + 64 hex chars = 32 bytes), but they are different things:

| Hash | Who generates it | Where to find it | HHP example |
|---|---|---|---|
| **UserOperation hash** | ZeroDev / Privy when building the UserOp | JiffyScan (if the bundler reports it) | `0xd6c061...` (NFT mint) |
| **Ethereum tx hash** | The Arbitrum node when confirming the tx | Arbiscan (search by Smart Wallet) | Generated by the bundler |
| **Transaction receipt hash** | Same as tx hash | Arbiscan | — |

#### The two hashes we found

- `0xd6c06101fb4a63f4789963908f5824cc8ee7f87aa9c92510f85899ad812360a6`
  → UserOp hash generated when the Smart Wallet minted the SpotNFT. ZeroDev returned it to the frontend as confirmation. The bundler wrapped that UserOp into a real Ethereum transaction that IS on-chain — but under a different hash.

- `0xf5c39060cb0e01a78bb077005fb03a2bb5b411e3a65c3e9d5c12dfe1b2309d57`
  → UserOp hash for the `deposit()` / "Pay my share" action. Same flow — the builder authorized the operation, ZeroDev executed it on-chain.

**Why Arbiscan can't find them:** Arbiscan indexes Ethereum transaction hashes. A UserOp hash is an internal identifier within the ERC-4337 system — it is not the tx hash. JiffyScan (a UserOp explorer) would look for them, but if ZeroDev's bundler doesn't report to its public pool, they won't appear there either.

**How to actually find the real transactions:** go to Arbiscan Sepolia → search the Smart Wallet address (`0x40c8aEc3...`) → look at its internal transaction history. You'll see the calls the EntryPoint executed on behalf of the Smart Wallet.

### Why `0x000...000` appears as "From" when minting

```
ERC-721 standard defines:
  Transfer(address indexed from, address indexed to, uint256 indexed tokenId)

A MINT is: Transfer(address(0), to, tokenId)

By convention, "from = address(0)" means creation —
it does not mean the token came from any real address.
```

In Solidity, `_mint(to, tokenId)` internally emits:
```solidity
emit Transfer(address(0), to, tokenId);
```

Arbiscan displays this literally: `From: 0x0000...0000`. This is the behavior defined in EIP-721, not a bug or anomaly. The same applies to ERC-20 (`_mint` also emits a Transfer from the zero address).

**In HHP specifically:** when the builder pays for their spot, the Escrow calls `spotNFT.mint(builder, bookingId)`. SpotNFT emits `Transfer(0x000...000, BuilderSmartWallet, tokenId)`. Arbiscan shows it as "minted to" the builder's Smart Wallet.

---

### Why SpotNFT doesn't appear verified on Arbiscan

SpotNFT and HackerHouseEscrow are deployed **dynamically** from inside a UserOperation:

```
EntryPoint.handleOps(UserOp)
    → executes Smart Wallet
        → Smart Wallet calls HackerHouseFactory.createHouse()
            → Factory deploys new HackerHouseEscrow(...)
            → Factory deploys new SpotNFT(...)
```

The deployment happens as an **internal transaction** inside a UserOperation. Arbiscan Sepolia does not index contracts deployed via internal calls from UserOps. This is why SpotNFT shows up as an unverified address or is not listed at all.

On **mainnet** this works correctly — block explorers index contracts deployed by any method.

## Full technical flow

### 1. Identity — Privy + ZeroDev (ERC-4337)

```
Builder signs up with email / wallet
    → Privy creates an Embedded Wallet (EOA) — encrypted key, never exposed on-chain
    → ZeroDev (Kernel) deploys a Smart Wallet linked to that EOA

Only the Smart Wallet appears on-chain.
The Embedded Wallet signs UserOperations in the background.
```

#### Data wallets — only ownership-proven wallets feed POAPs/gates

A builder can add extra wallets as **read-only data sources** (they aggregate POAPs/credentials). For integrity, those wallets only count if the user **proved ownership** by signing via Privy `linkWallet` (reconciled server-side against `linked_accounts`). There is no raw-address text input. A wallet already registered to another user cannot be reused (global anti-reuse). POAP sync iterates only the primary wallet plus `verified` wallets. Full detail in [`docs/features/link-wallet.md`](./features/link-wallet.md#secure-wallet-linking-2026-06-14).

### 2. House creation — HackerHouseFactory

When the host creates a Hacker House, the frontend calls `HackerHouseFactory.createHouse()`. The Factory deploys **3 contracts in a single transaction**:

```
HackerHouseFactory.createHouse(...)
    │
    ├── 1. [optional] MockYieldAdapter → if the house uses yield (GMX mode)
    │
    ├── 2. HackerHouseEscrow → the contract that holds the funds
    │        - USDC token address
    │        - hostSafe (host's multisig)
    │        - depositAmount per spot
    │        - withdrawDate (when the host can withdraw)
    │        - capacity (available spots)
    │
    ├── 3. SpotNFT → ERC-721, one per house
    │        - Only the escrow can mint / burn
    │        - On-chain metadata (Base64 JSON embedded in Solidity)
    │
    └── 4. escrow.initialize(spotNFT) → links them together
```

**Verified contracts on Arbitrum Sepolia:**
- Factory: [`0x751ea80f...`](https://sepolia.arbiscan.io/address/0x751ea80fae2f714812bf0317be4df96fd3ffcfb5#code)
- MockUSDC: [`0x999579cc...`](https://sepolia.arbiscan.io/address/0x999579cc79400a1b59b119b6697664dd9122ad93#code)

### 3. Paying for a spot — HackerHouseEscrow.deposit()

```
Builder clicks "Pay"
    │
    ├── Smart Wallet approves USDC to the escrow (approve)
    ├── Smart Wallet calls escrow.deposit(bookingId)
    │
    └── Escrow:
         ├── Receives USDC from the builder
         ├── Records deposit: deposits[builder] = amount
         ├── Marks hasDeposited[builder] = true
         ├── If yieldMode == GMX → deposits into YieldAdapter
         └── Calls spotNFT.mint(builder, bookingId)
              └── Builder receives NFT in their Smart Wallet
                  └── Arbiscan shows: Transfer(0x000...000, SmartWallet, tokenId)
```

**On-chain result:**
- NFT `HHP Spot #0` in the builder's wallet
- USDC locked in the escrow
- Nobody — not the host, not HHP — can touch those funds until `withdrawDate`

---

### 4. Release — escrow.release()

```
After withdrawDate:
    HostSafe calls escrow.release()
        │
        ├── Calculates fee: 0.5% of total → HHP Treasury
        ├── If yield mode:
        │    ├── Withdraws from YieldAdapter
        │    └── Distributes yield to HOST or to BUILDERS
        │
        ├── Transfers (total - fee) → hostSafe
        └── Transfers fee → HHP_TREASURY (0xd7ed1a...)
```

---

### 5. Cancellation — escrow.cancel()

```
HostSafe calls escrow.cancel()
    │
    └── For each depositor in _depositors[]:
         ├── Withdraws USDC from YieldAdapter (if applicable)
         ├── Transfers deposit back to builder
         └── Burns their SpotNFT (spotNFT.burn)
```

Auto-refund — builders don't need to claim anything.

---

### 6. The NFT

```solidity
constructor(address _escrow, string memory _houseName) ERC721("HHP Spot", "SPOT")
```

- **One collection per house** — each HackerHouseEscrow has its own SpotNFT contract
- **On-chain metadata** — generated in Solidity with Base64, no IPFS or server dependency
- **Only the escrow can mint/burn** — no one can forge a spot
- **Token ID = bookingId** — `#0` is the first spot, `#1` the second, etc.

## Why this matters (pitch)

| Without HHP | With HHP |
|---|---|
| Host asks for a wire transfer and you trust them | Funds in escrow — trustless |
| Builder loses money if host cancels | Auto-refund by contract |
| Platform holds your data and funds | Smart wallet, protected identity |
| PDF as confirmation | On-chain NFT, verifiable by anyone |
| Centralized | Arbitrum L2 — cheap, fast, auditable |

**0.5% fee** on release — HHP charges those who want access to builders, not the builders themselves.
